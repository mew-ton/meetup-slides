## Google Chrome

2026/06/02

![bg 15%](./_assets/logos/googlechrome.png)

---

## "Google Chrome" とは

- Google が開発する, デスクトップで圧倒的シェアを持つ Web ブラウザ
- レンダリングエンジンは Blink, 新しい Web 機能の実装を最も積極的に進める
- Edge など他の Chromium 系ブラウザの基盤にもなっている

---

## Google Chrome のニュース

**Chrome 149 で CSS "gap decorations" が安定版に**

- グリッド / フレックスの **gap (余白) に区切り線を引ける** CSS 新機能
- `column-rule` を拡張し, 行方向用の `row-rule` を新設
- **Chromium 系 (Chrome / Edge) が単独先行, Safari / Firefox は未対応**
- 非対応ブラウザでは線が出ないだけ → プログレッシブエンハンスメント可

<!-- _footer: "出典: [Gap decorations: Now available in Chromium (Chrome for Developers)](https://developer.chrome.com/blog/gap-decorations-stable)" -->

---

## 何が嬉しいのか

```css
.grid {
  display: grid;
  gap: 2rem;
  column-rule: 1px solid #ddd; /* 列の間に縦線 */
  row-rule: 2px solid #000; /* 行の間に横線 */
}
```

- 従来は **区切り線のために border + 疑似要素 + 計算** が必要だった
- gap の値が変わっても線は自動で追従するので保守が楽
- テーブル風・ノート風レイアウトが数行で書ける

## npm

2026/06/09

![bg 15%](./_assets/logos/npm.png)

---

## "npm" とは

- Node.js 標準のパッケージマネージャ / 世界最大の JS パッケージレジストリ
- `npm install` 一発で依存を解決する, フロントエンド開発の生命線
- 現在は GitHub (Microsoft) が運営している

---

## npm のニュース

**npm v12 で install スクリプトの自動実行を既定で無効化**

- `preinstall` / `install` / `postinstall` を **デフォルトで実行しない** 方針へ
- 実行を許すパッケージだけ **allowlist で明示許可** (`npm approve-scripts`)
- 近年多発する **postinstall 経由のサプライチェーン攻撃** への直接的な対策
- 2026 年 7 月リリース予定, CI の移行が必要

<!-- _footer: "出典: [Upcoming breaking changes for npm v12 (GitHub Changelog)](https://github.blog/changelog/2026-06-09-upcoming-breaking-changes-for-npm-v12/)" -->

---

## なぜ大きいのか

- これまで `npm install` は依存パッケージの **任意コードを自動実行** していた
- これが「入れた瞬間に感染する」サプライチェーン攻撃の温床だった
- v12 以降は **「入れる」と「実行する」を分離**, 既定で安全側に倒れる
- これまで手動で付けていた `--ignore-scripts` が, ついに **デフォルト** になる

- Evan You は引き続き OSS のロードマップを主導, **ベンダー中立を維持** すると表明

## polyfill.io

2026/05/22

---

## "polyfill" とは

- 新しいブラウザ機能を, 未対応の古いブラウザでも動くよう補う互換コード
- 「欠けている機能を埋める (fill in)」ためのもの
- かつては IE 対応などで広く使われた

---

## "polyfill.io" とは

- 必要な polyfill だけを判定し, CDN から自動配信していたサービス
- `<script src="https://polyfill.io/...">` の 1 行で使えるお手軽さで普及
- 一時は 10 万以上のサイトが読み込む定番サービスだった

---

## 2024 年に何が起きていたか

- 2 月, 運営元が **中国企業 (Funnull)** に交代し, 信頼できない手に渡った
- 交代後, **特定の条件下でのみ悪意あるサイトへ誘導** する挙動が報告される
  - 時間帯や端末を絞り管理者を避ける **ステルス的** な手口で, 大きくは表沙汰にならず
- 6 月にドメインは停止されたが, **`<script>` 参照を消し忘れたサイトが多数残存**

---

## polyfill.io のインシデント

**続報: 放置された参照が原因で, 不審なログイン画面が続出**

- 5/22 (JST) にドメインが Namecheap → GoDaddy へ移管され, 再び応答を開始
- 旧 polyfill.io への参照が残るページで **HTTP 401 (BASIC 認証) ダイアログ** が出現
- **東芝・無印良品 など複数の国内企業** で報告が相次ぎ, 各社が注意喚起

<!-- _footer: "出典: [東芝や無印良品など、複数の企業で「不審なログイン画面」(ITmedia NEWS)](https://www.itmedia.co.jp/news/articles/2606/04/news102.html) / [放置サイトで不審なダイアログ (Qiita)](https://qiita.com/Robot-Inventor/items/2f32a66f087a3bf7eaa5)" -->

---

## 今回の意味

- 直接の実害は認証ダイアログだが, **2024 年の "置き土産" が今も生きている** ことを示した
- 第三者ドメインを **信頼し続けている** こと自体がリスクだと再確認させられた

---

## 教訓 — 外部スクリプトを放置しない

- 使わなくなった外部スクリプトの参照は **確実に削除する**
- CDN 配信の polyfill は **self-host** に切り替えるか, 不要なら外す
- `Content-Security-Policy` で読み込み先ドメインを制限しておく
- 「入れて終わり」ではなく, 依存と参照を **棚卸しし続ける**

# 特別企画 — フロントエンドの「担い手」の歴史

フロントエンド エンジニア集会 2026 年 6 月

---

## 今回の特別企画とは

**フロントエンドを "誰が" 支えてきたのか** を俯瞰します.

---

## jQuery (2006)

- ブラウザ間の差異の吸収が最大の課題だった時代を制した定番ライブラリ
- John Resig が **個人** で公開し, コミュニティが育てた
- のちに **OpenJS Foundation** が管理する公的資産へ

---

## Angular (2010)

- **Google** が開発するフルスタックなフレームワーク (AngularJS → Angular)
- 大企業が一貫して主導し続ける **安定したガバナンス** の代表例
- 担い手が変わらないまま, 着実に進化を続けている

---

## React (2013)

- **Facebook (Meta)** が公開した UI ライブラリ, コンポーネント指向を普及
- 企業がフレームワークを主導する時代を象徴する存在に
- **2026 年, Meta を離れ React Foundation へ** (企業 → 中立な財団)

---

## Vue (2014)

- **Evan You が個人** で開発, スポンサー / コミュニティで運営
- 特定の大企業に依存しない **独立系フレームワークの代表格**
- 今もコミュニティ主導のガバナンスを保っている

---

## Next.js (2016)

- **Vercel** が開発する React ベースのメタフレームワーク
- フレームワークとホスティング (Vercel) を **セットで提供**
- React 人気の流入を **インフラ利用ごと抱え込む** 商業モデルを確立

---

## Svelte (2016)

- **Rich Harris が個人** で生み出した, 仮想 DOM を使わない新発想
- コンパイラ方式で軽量・高速, 熱心なコミュニティを獲得
- 作者が **Vercel に入社 (2021)**, 以後は Vercel が開発を支える

---

## Vite (2020)

- **Evan You が再び個人発** で作った高速ビルドツール (週 約 1.3 億 DL)
- フレームワーク非依存で, React / Vue / Svelte などの **共通土台** に
- VoidZero を経て **2026 年 Cloudflare へ** (個人 → 企業)

---

## Astro (2021)

- コンテンツ重視のサイト向け高速フレームワーク (Fred Schott ら)
- **2026 年 1 月, Cloudflare が買収** (OSS は継続)
- Vite と合わせ, **Cloudflare がフレームワーク〜配信を垂直統合**

<!-- _footer: "出典: [Cloudflare Acquires Team Behind Astro (The New Stack)](https://thenewstack.io/cloudflare-acquires-team-behind-open-source-framework-astro/)" -->

---

## Remix (2021)

- React 向けのメタフレームワーク (Ryan Florence ら)
- Web 標準に寄せた設計で注目を集めた
- **2022 年 Shopify が買収**, 企業の傘下で開発が進む

---

## Vercel vs Cloudflare

| 層             | Vercel           | Cloudflare      |
| -------------- | ---------------- | --------------- |
| フレームワーク | Next.js / Svelte | Astro           |
| ビルド基盤     | Turbopack        | Vite (VoidZero) |
| インフラ       | Vercel           | エッジ / Void   |

---

## Vercel vs Cloudflare

- どちらも **フレームワーク + ビルド + インフラ** を垂直に束ねてきた
- Cloudflare 側の **Void** は `void deploy` だけで DB / 認証 / AI まで自動構築する
  Vite ネイティブのデプロイ基盤 (プレビュー段階)
- フロントエンドの主導権争いは, **プラットフォーム企業同士の対決** へ

---

## 今後, 誰が中核を握るのか

- **Vercel** は React 人気を背景に, フレームワーク + ホスティングで先行
- **Cloudflare** は Astro / Vite を買収し, エッジ配信から対抗
- 一方で React は **財団へ独立**, 中立であろうとする流れも残る
- 我々が選ぶ「フレームワーク」は, 裏で **どの陣営に乗るか** の選択でもある
