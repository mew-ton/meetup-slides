## CSS gap decorations

2026/06/02

![bg 15%](./_assets/logos/css.png)

---

## "CSS gap decorations" とは

- グリッド / フレックス / マルチカラムの **gap (余白) に区切り線を引く** 新仕様
- 既存の `column-rule` を拡張し, 行方向用の `row-rule` を新設
- これまで border や疑似要素のハックが必要だった処理を CSS だけで実現

---

## CSS gap decorations のニュース

**Chrome 149 で gap decorations が安定版に**

- `column-rule` が grid / flexbox / multicol でも使えるように
- 行の区切りを描く `row-rule` が新登場
- `rule-inset` や `*-rule-visibility-items` で線の伸縮・表示位置を制御
- 非対応ブラウザでは線が出ないだけ → プログレッシブエンハンスメント可

<!-- _footer: "出典: [Gap decorations: Now available in Chromium (Chrome for Developers)](https://developer.chrome.com/blog/gap-decorations-stable)" -->

---

## 何が嬉しいのか

```css
.grid {
  display: grid;
  gap: 2rem;
  column-rule: 1px solid #ddd; /* 列の間に縦線 */
  row-rule: 2px solid #000;    /* 行の間に横線 */
}
```

- 従来は **区切り線のために border + 疑似要素 + 計算** が必要だった
- gap の値が変わっても線は自動で追従するので保守が楽
- テーブル風・ノート風レイアウトが数行で書ける

## Safari

2026/06/08

![bg 15%](./_assets/logos/safari.png)

---

## "Safari" とは

- Apple の WebKit エンジンを採用した Web ブラウザ
- iOS / iPadOS では長らく唯一許可されてきたブラウザエンジン
- 毎年 WWDC で, 秋に来る次期 Safari のベータと新機能が発表される

---

## Safari のニュース

**WWDC26 で Safari 27 beta を公開 (58 の新機能 / 525 の修正)**

- **Customizable Select** — `<select>` を JS なしでフルにスタイル可能, アクセシビリティも維持
- **Scroll Anchoring** — ビューポート上にコンテンツが挿入されてもガタつかない
- `stretch` キーワード, `img sizes=auto`, Grid Lanes, HTML `<model>` 要素 など

<!-- _footer: "出典: [News from WWDC26: WebKit in Safari 27 beta (WebKit)](https://webkit.org/blog/17967/news-from-wwdc26-webkit-in-safari-27-beta/)" -->

---

## Customizable Select の意味

- これまで `<select>` の見た目は **OS / ブラウザ任せ** で, 凝った UI を作るには
  JS で独自実装するしかなかった (= アクセシビリティを犠牲にしがち)
- ネイティブ要素のまま自由にスタイルできる → **車輪の再発明が要らなくなる**
- Chrome が先行実装していた機能に Safari も追従 → **各ブラウザが揃ってきた**

## "Red Hat" とは

- エンタープライズ Linux で知られる Red Hat (IBM 傘下) が公開する npm パッケージ群
- `@redhat-cloud-services` スコープで, 管理コンソール向けの
  **フロントエンドコンポーネント / API クライアント / 開発ツール** を配布

## Red Hat のインシデント

**`@redhat-cloud-services` の 32 パッケージが侵害される**

- 6/1 に 32 パッケージ / 96 バージョンへ悪意あるコードが混入 (週 11.7 万 DL 規模)
- `preinstall` スクリプトで認証情報窃取マルウェア **"Miasma"** が自動実行
- GitHub Actions の **OIDC トークンを悪用** して正規スコープから publish (CI 侵害)

<!-- _footer: "出典: [Miasma: Supply Chain Attack Targeting RedHat npm Packages (Wiz)](https://www.wiz.io/blog/miasma-supply-chain-attack-targeting-redhat-npm-packages) / [RHSB-2026-006 (Red Hat)](https://access.redhat.com/security/vulnerabilities/RHSB-2026-006)" -->

---

## 先月の続き — Mini Shai-Hulud の新亜種

- "Miasma: The Spreading Blight" は **5 月の Mini Shai-Hulud と同系統** のワーム
- 手口も同じ: **OIDC トークンの窃取** で信頼された namespace に成りすまし
- 個人の認証情報ではなく **CI の信頼関係** を突くため, 多要素認証では防げない
- 「信頼された大手の namespace」でも安全とは限らない, という事例が続いている

## Microsoft

2026/05/29

![bg 15%](./_assets/logos/npm.png)

---

## "Microsoft" とは

- Windows / Azure / GitHub などを擁する巨大テック企業
- **Microsoft Security / MSRC** として脅威インテリジェンスを継続的に公開
- npm エコシステムを狙う攻撃キャンペーンの調査・注意喚起も行っている

---

## Microsoft のインシデント

**dependency confusion を悪用した 33 個の悪意ある npm パッケージを報告**

- 5/28〜29 に, **社内向けパッケージになりすました** 33 個が npm に公開される
- `package.json` に GitHub Enterprise / Jira など **社内インフラの URL を偽装**
- インストールすると C2 から難読化された **偵察ペイロード** を取得・実行

<!-- _footer: "出典: [Malicious npm packages abuse dependency confusion to profile developer environments (Microsoft Security Blog)](https://www.microsoft.com/en-us/security/blog/2026/05/29/33-malicious-npm-packages-abuse-dependency-confusion-profile-developer-environments/)" -->

---

## dependency confusion とは

- 社内 private registry にある **内部パッケージと同名** のものを公開 npm に置く手口
- 設定次第で `npm install` が **公開側 (攻撃者) のパッケージを優先** してしまう
- 今回は 9 つの組織スコープを標的に, **どの企業が引っかかるかを偵察** していた

---

## polyfill.io

2026/05/22

---

## "polyfill.io" とは

- 古いブラウザに不足機能 (Polyfill) を CDN から自動配信していたサービス
- かつては多くのサイトが `<script src="https://polyfill.io/...">` の 1 行で利用
- 2024 年に所有者が変わり, マルウェアを配信する **サプライチェーン攻撃** の舞台に

---

## polyfill.io のインシデント

**放置された参照が原因で, 不審なログイン画面が続出**

- 5/22 (JST) にドメインが Namecheap → GoDaddy へ移管, 再び応答を開始
- 旧 polyfill.io への参照が残るページで **HTTP 401 (BASIC 認証) ダイアログ** が出現
- 東芝・無印良品・ディスクユニオン・Yahoo! JAPAN など国内大手で報告が相次ぐ

<!-- _footer: "出典: [Polyfill.ioを放置したサイトで不審なダイアログが表示されている件 (Qiita)](https://qiita.com/Robot-Inventor/items/2f32a66f087a3bf7eaa5)" -->

---

## 何が起きていたのか

- 2024 年の攻撃後, polyfill.io は危険として各所で利用停止が呼びかけられた
- しかし **`<script>` 参照を消し忘れた** サイトが大量に残存していた
- ドメインが再活性化して 401 を返すと, ブラウザが認証ダイアログを表示する
- 今回の直接の実害は認証ダイアログだが, **第三者ドメインを信頼し続けている** こと自体がリスク

---

## 我々はどう守るか

- 依存追加・更新時は **新しいバージョンを即取り込まない** (数日寝かせる)
- `npm install --ignore-scripts` で **pre/postinstall スクリプトを無効化**
- `.npmrc` で社内スコープを private registry に固定 (**dependency confusion 対策**)
- 使わない外部スクリプト参照は確実に削除し, **CSP** で読み込み先を制限
- 侵害が疑われたら **CI シークレット / npm トークンを即ローテーション**

# 特別企画 — フロントエンドの「担い手」の歴史

## 今回の特別企画とは

今月は VoidZero (Vite) の Cloudflare 買収がありました.
一方で 2 月には React が Meta を離れて **React Foundation** へ.
これを機に, **フロントエンドを "誰が" 支えてきたのか** を俯瞰します.

---

## VoidZero

2026/06/03

![bg 15%](./_assets/logos/voidzero.png)

---

## "VoidZero" とは

- Vue / Vite の作者 **Evan You** が 2024 年に設立した企業
- Vite / Vitest / Rolldown / Oxc を統合ツールチェーン **"Vite+"** としてまとめる
- Vite は週 **約 1.3 億 DL**, 現代フロントエンドのビルド基盤の事実上の標準

---

## VoidZero のニュース

**Cloudflare が VoidZero を買収**

- チーム全員が Cloudflare へ. Vite / Vitest / Rolldown / Oxc は **OSS のまま維持**
- VoidZero が作っていた Vite ネイティブのデプロイ基盤 **Void** ごと取り込み
- 背景: AI が生成するアプリは **既定で Vite テンプレート**, デプロイ先は Cloudflare

<!-- _footer: "出典: [VoidZero is Joining Cloudflare (VoidZero)](https://voidzero.dev/posts/voidzero-cloudflare)" -->

---

## 黎明期 — "ライブラリ" の時代 (〜2000 年代)

- 1995 年 **JavaScript** 誕生, その後ブラウザ戦争で実装差が乱立
- 課題は「ブラウザ間の差異をどう吸収するか」だった
- 2006 **jQuery** (John Resig, 個人) が定番化 → のちに OpenJS Foundation へ

---

## フレームワーク戦国時代 (2010 年代)

- 2010 **AngularJS** (Google), 2013 **React** (Facebook / Meta)
- 2014 **Vue.js** (Evan You, 個人), 2016 **Svelte** (Rich Harris, 個人)
- 仮想 DOM / コンポーネント指向で **SPA** が主流に, 担い手は企業と個人が混在

---

## ビルドツールとメタフレームワーク (2015〜)

- ES2015 以降, **webpack / Babel** をコミュニティ + スポンサーが支える
- 2016 **Next.js** (Vercel) — フレームワークとホスティングが一体化
- 2020 **Vite** (Evan You) / **esbuild** (Figma の Evan Wallace) が高速化を牽引

---

## フレームワーク + インフラの囲い込み

- **Next.js + Vercel** はフレームワークとインフラがセット
  → React 人気の流入を **インフラ利用ごと** 抱え込む構造
- **Vite + Void (VoidZero)** はフレームワーク非依存で土台を押さえ, インフラも狙う
  → そこを **Cloudflare が買収**, React / Vercel 圏に対抗する構図に

---

## Cloudflare の 2026

**1 年で "フロントエンドの主要ピース" を立て続けに買収**

- 1 月: **Astro** (コンテンツ向けフレームワーク) を買収
- 6 月: **VoidZero** (Vite ツールチェーン + デプロイ基盤 Void) を買収
- フレームワーク → ビルド基盤 → エッジ配信まで **一気に垂直統合**

<!-- _footer: "出典: [Cloudflare Acquires Team Behind Astro (The New Stack)](https://thenewstack.io/cloudflare-acquires-team-behind-open-source-framework-astro/)" -->

---

## 所有権は一方向ではない

| 時期 | 技術 | 動き |
|---|---|---|
| 2026/02 | **React** | Meta → React Foundation (企業 → 中立な財団) |
| 2026/06 | **Vite** | Evan You → Cloudflare (個人 → 企業) |

- 同じ年に **逆向き** の動きが起きた
- 企業が手放して中立化する流れと, 個人発が企業へ集約される流れが同居

## まとめ — 誰が中核を握るのか

- フロントエンドの担い手は **個人 / 企業 / 財団** を行き来してきた
- **中立化 (React → 財団)** と **集約 (Astro・Vite → Cloudflare)** が同時に進む 2026 年
- OSS は維持されても, **方向性を誰が決めるのか** は静かに移り変わっていく
