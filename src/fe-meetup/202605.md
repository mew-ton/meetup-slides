## "TypeScript" とは

- Microsoft が開発する, JavaScript に静的型付けを加えた言語
- ほぼすべてのモダンフロントエンドで採用される事実上の標準

**TypeScript 6.0 が正式リリース**

- 6.1 / 6.2 はリリースされず, **次は 7.0** へ直接ジャンプの予定

## ECMAScript 2026

2026/04 (RC 公開)

![bg 15%](./_assets/logos/ecmascript.png)

---

## "ECMAScript" とは

- JavaScript のコア仕様を定める標準規格
- TC39 が策定し, 毎年 6 月頃に最終版が承認される年次リリース
- 各ブラウザや Node.js のランタイムはこの仕様を実装する

---

## ECMAScript 2026 のニュース

**ES2026 RC が公開, Temporal が Stage 4 入り**

- **Temporal API** が Stage 4 → ES2026 仕様に正式採用
  - `Date` の代替, イミュータブル, タイムゾーン / 暦をネイティブサポート
- `JSON.parse` の **source text アクセス** (BigInt 安全な数値復元)
- `Iterator.concat()` ほか, **イテレータ系メソッドの拡充**
- `Array.fromAsync` も標準入り

<!-- _footer: "出典: [TC39 Advances Temporal to Stage 4](https://socket.dev/blog/tc39-advances-temporal-to-stage-4)" -->

## axios

2026/03/31

![bg 15%](./_assets/logos/npm.png)

---

## "axios" とは

- ブラウザ / Node.js で動く HTTP クライアントライブラリ
- Promise ベースの直感的な API で, fetch 以前から広く使われてきた定番
- npm の週間ダウンロード数は **約 7,000 万** という巨大パッケージ

---

## axios のインシデント

**メンテナアカウントが乗っ取られ, 悪意のあるバージョンが公開**

- 3/31 に `1.14.1` / `0.30.4` の 2 バージョンが npm 上に出現
- `plain-crypto-js@4.2.1` を依存に注入し, **RAT (Remote Access Trojan)** を配布
- macOS / Windows / Linux いずれにも対応する仕掛け
- 公開からおよそ **3 時間で削除** されたが影響範囲は甚大

<!-- _footer: "出典: [Mitigating the Axios npm supply chain compromise (Microsoft)](https://www.microsoft.com/en-us/security/blog/2026/04/01/mitigating-the-axios-npm-supply-chain-compromise/)" -->

---

## 何が起きたのか

- メンテナ `jasonsaayman` の PC が **標的型ソーシャルエンジニアリング + RAT** で侵入
- npm 認証情報を奪い, メールアドレスを攻撃者の Proton Mail に変更
- そのまま悪意のあるバージョンを公開

**1 人のメンテナの PC が落ちると, 世界中の CI が壊れる**

---

## 攻撃者の正体

- Microsoft Threat Intelligence は **Sapphire Sleet (北朝鮮系)** に帰属
- Google GTIG は **UNC1069** として追跡 (2018 年から活動)
- 国家レベルのアクターが OSS の **メンテナ個人** を直接狙う時代に

- TanStack Query (旧 React Query), Router, Table などを擁する人気ライブラリ群

## TanStack のインシデント

**42 個の `@tanstack/*` パッケージに悪意のあるバージョンが公開**

- 5/11 19:20〜19:26 UTC のわずか **6 分間** で 84 バージョンが一斉公開
- `@mistralai/*`, `@squawk/*`, OpenSearch, Guardrails AI, UiPath なども巻き込み
- 最終的に **170+ パッケージ / 400+ 悪意バージョン** に拡大
- 外部研究者により 20〜26 分で検知 → 全バージョン deprecate, npm 側で tarball 削除

## 攻撃手法: Mini Shai-Hulud

- GitHub Actions の **`pull_request_target` ("Pwn Request") パターン** を悪用
- フォークとベースの信頼境界をまたぐ **Actions キャッシュポイズニング**
- ランナープロセスのメモリから **OIDC トークンを抜き取り**, npm 連携を奪取
- パスワードや token を盗むのではなく, **CI の信頼関係そのもの** を悪用

---

## "ワーム" としての怖さ

- 盗んだ npm トークン / GitHub PAT で **被害者が書き込み権を持つ別パッケージ** にも伝播
- npm / PyPI / Composer を **横断** して自己複製
- 開発者マシンには `gh-token-monitor` を常駐, 60 秒ごとに GitHub をポーリング
- token 失効 (40X) を検知すると **`rm -rf ~/` を試みる** デッドマンスイッチ付き

---

## 史上初の "正規署名された" 悪意パッケージ

- 侵害版は **SLSA Build Level 3 の provenance attestation を正しく持っていた**
- 「ビルドの出所が検証できる」仕組みを潜り抜けた最初の事例
- "署名されているから安全" は崩れた

## なぜ "OSS 化" するのか

- 攻撃のコモディティ化 → 模倣犯が増えれば **検知者側の負荷が爆発**
- 自分たちの攻撃が "1 件" ではなく "N 件" に化ける
- 防御側にとっては, **同じ手口の亜種が無数に飛んでくる** 最悪の状況

---

## 我々はどう守るか

- `package-lock.json` / `yarn.lock` のコミットは前提として
- 依存追加時には **新しいバージョンを即座に取り込まない** (数日寝かせる)
- CI の GitHub Actions では `pull_request_target` を **原則禁止**
- `npm install` 時の **postinstall / prepare スクリプトを無効化** する選択肢 (`--ignore-scripts`)
- 開発機と本番環境のクレデンシャルを **物理的に分離**
